# 常熟地图旅行规划 Web 第一版本开发方案

## 1. 项目定位

本项目第一版本要做一个以「常熟市地图」为入口的本地旅行规划 Web。

用户进入页面后，首先看到常熟市地图。地图上用小坐标点展示重点景点、非遗店、美食与美食店铺。用户点击坐标点后出现对应卡片，并可以把卡片拖入右侧「行程卡片槽」中，形成自己的旅行计划。

第一版本重点不是做一个完整的商业旅游平台，而是做出一个可演示、可交互、可扩展的原型产品：

- 地图是主要入口。
- 点位是发现内容的方式。
- 卡片是承载信息的核心组件。
- 拖拽是用户规划路线的主要动作。
- J 人 / P 人模式是产品差异点。

## 2. 第一版本目标

第一版本需要实现以下核心体验：

1. 页面打开后直接进入常熟市地图界面。
2. 地图上显示多个小坐标点，不同类型点位有不同视觉标识。
3. 单击地图点位后，展示对应的简化信息卡片。
4. 卡片可以拖动到右侧卡片槽，用于规划想去的地方。
5. 右侧卡片槽可以保存用户选择的地点，并显示路线顺序。
6. 用户可以切换 J 人模式和 P 人模式。
7. J 人模式下，用户主要通过手动拖拽规划路线。
8. P 人模式下，系统可以自动生成或随机生成路线，并把路线卡片放入右侧卡片槽。
9. 双击卡片后，卡片从简化状态变为详细状态。
10. 详细状态支持下拉查看更完整的介绍内容。

## 3. 第一版本不做的内容

为了保证第一版本可以快速落地，以下内容暂不作为 V1 必做项：

- 不做真实导航级路径计算。
- 不做实时人流数据接入。
- 不做真实门票接口接入。
- 不做真实点评平台接口接入。
- 不做用户登录系统。
- 不做支付、预约、购票功能。
- 不做复杂后台管理系统。
- 不做完全准确的商业 POI 数据库。

第一版本中的开放时间、门票价位、人流量估计、点评等内容以本地静态数据为主，字段结构先设计完整，具体内容后续可替换为真实数据。

## 4. 页面整体布局

### 4.1 页面结构

页面采用单页应用结构，首屏即主功能界面。

整体布局分为三块：

- 顶部工具栏
- 左侧 / 中间地图区域
- 右侧行程卡片槽

推荐桌面端布局：

```text
┌──────────────────────────────────────────────────────────────┐
│ 顶部工具栏：类型筛选 / J人模式 / P人模式 / 生成路线 / 清空路线 │
├──────────────────────────────────────────────┬───────────────┤
│                                              │               │
│                                              │  我的行程      │
│               常熟市地图                     │  卡片槽        │
│                                              │               │
│                                              │               │
└──────────────────────────────────────────────┴───────────────┘
```

移动端可以改为：

- 地图占据主要屏幕。
- 行程卡片槽变成底部抽屉。
- 点位详情卡片以底部弹层形式出现。

第一版本可以优先保证桌面端展示效果，再兼顾移动端基础可用。

### 4.2 顶部工具栏

顶部工具栏包含：

- 类型筛选：全部、景点、非遗、美食、美食店铺
- 模式切换：J 人模式、P 人模式
- P 人按钮：生成推荐路线、随机路线
- 行程操作：清空行程
- 可选信息：当前已选地点数量、预计游玩时长

## 5. 地图区域设计

### 5.1 地图呈现方式

第一版本建议采用以下两种方案之一：

方案 A：Leaflet 地图方案

- 使用 Leaflet 加载在线地图瓦片。
- 在常熟市区域上放置点位 Marker。
- 优点是真实地图感强，点位逻辑自然。
- 缺点是需要网络地图瓦片，视觉风格受地图服务影响。

方案 B：自定义常熟地图视觉方案

- 使用一张常熟市地图底图，或者使用简化的 SVG / Canvas 风格地图。
- 在地图容器上按相对坐标放置点位。
- 优点是视觉可控，适合黑客松展示。
- 缺点是不是真实导航地图。

V1 推荐：

优先使用「自定义常熟地图视觉方案」或「Leaflet + 静态点位」。如果开发时间紧，先用自定义地图视觉方案，确保交互完整。

### 5.2 点位展示

地图初始状态只显示小坐标点，不直接铺开所有卡片，避免页面拥挤。

点位类型：

- 景点：使用绿色或山形/地标类图标。
- 非遗：使用蓝色或手作/印章类图标。
- 美食：使用橙色或碗筷类图标。
- 美食店铺：使用红色或店铺类图标。

点位状态：

- 默认状态：小圆点或图标。
- 悬浮状态：显示名称 Tooltip。
- 单击状态：弹出简化卡片。
- 已加入行程状态：点位高亮，并显示序号。

### 5.3 地图上的路线表现

当用户把多个卡片拖入右侧行程槽后，地图上可以用线连接这些地点，形成一个简单路线。

V1 路线表现不要求真实道路，只需要按用户选择顺序连接点位即可。

地图路线规则：

- 第 1 个地点显示序号 1。
- 第 2 个地点显示序号 2。
- 依次类推。
- 地图上用折线连接这些点。
- 删除或调整右侧卡片顺序后，地图路线同步更新。

## 6. 卡片系统设计

卡片是本项目最核心的 UI 元素。

### 6.1 卡片状态

卡片分为两种状态：

简化状态：

- 用于快速了解地点。
- 适合在地图点位点击后弹出。
- 适合在右侧卡片槽中展示。
- 信息短，不占太多空间。

详细状态：

- 用户双击卡片后进入详细状态。
- 卡片变宽、变长。
- 内容区域可滚动。
- 展示完整介绍、开放时间、价格、人流、点评等信息。

### 6.2 卡片交互

卡片需要支持以下交互：

- 单击地图点位：打开简化卡片。
- 点击卡片上的加入按钮：加入右侧行程槽。
- 拖拽卡片：拖入右侧行程槽。
- 双击卡片：进入详细状态。
- 点击展开按钮：进入详细状态，作为双击的补充入口。
- 点击收起按钮：回到简化状态。
- 在右侧卡片槽中拖动卡片：调整行程顺序。
- 点击删除按钮：从行程中移除。

### 6.3 景点卡片字段

景点卡片需要包含：

- 景点名称
- 景点类型
- 一句话简介
- 开放时间
- 景区门票价位
- 景区人流量估计
- 景点历史简介
- 推荐游玩时长
- 适合人群
- 地图位置

简化状态展示：

- 景点名称
- 类型标签
- 一句话简介
- 人流量估计
- 推荐游玩时长

详细状态展示：

- 景点名称
- 开放时间
- 门票价位
- 人流量估计
- 历史简介
- 推荐游玩方式
- 注意事项

### 6.4 非遗卡片字段

非遗卡片与景点卡片结构基本一致，但文案重点改为非遗体验。

非遗卡片需要包含：

- 非遗项目或店铺名称
- 非遗类型
- 一句话简介
- 开放时间
- 门票或体验价位
- 人流量估计
- 非遗历史简介
- 可体验内容
- 推荐停留时长
- 适合人群
- 地图位置

简化状态展示：

- 非遗名称
- 类型标签
- 一句话简介
- 人流量估计
- 推荐停留时长

详细状态展示：

- 非遗名称
- 开放时间
- 体验价位
- 人流量估计
- 非遗历史简介
- 可体验项目
- 预约提示

### 6.5 美食卡片字段

美食卡片分为两类：

- 具体美食卡片
- 具体美食店铺卡片

#### 6.5.1 具体美食卡片

具体美食卡片展示某一种常熟美食本身，例如某个地方特色菜、小吃或传统食品。

需要包含：

- 美食名称
- 美食类别
- 一句话简介
- 美食历史简介
- 口味特点
- 推荐品尝场景
- 推荐相关店铺

简化状态展示：

- 美食名称
- 类型标签
- 一句话简介
- 口味特点

详细状态展示：

- 美食名称
- 美食历史简介
- 口味特点
- 相关故事
- 推荐搭配
- 关联店铺列表

#### 6.5.2 具体美食店铺卡片

具体美食店铺卡片展示某个店铺本身。

需要包含：

- 店铺名称
- 店铺类型
- 主打美食
- 一句话简介
- 营业时间
- 人均消费
- 店铺火热情况估计
- 店铺点评
- 推荐菜品
- 地图位置

简化状态展示：

- 店铺名称
- 主打美食
- 火热情况估计
- 人均消费

详细状态展示：

- 店铺名称
- 营业时间
- 主打美食
- 人均消费
- 火热情况估计

## 7. 行程卡片槽设计

右侧卡片槽用于承载用户规划结果。

### 7.1 基础功能

行程卡片槽需要支持：

- 接收从地图卡片拖入的地点。
- 展示已选卡片列表。
- 显示地点顺序。
- 支持拖动排序。
- 支持删除单个卡片。
- 支持清空全部。
- 显示预计游玩总时长。
- 显示当前路线类型：手动规划 / 系统生成 / 随机路线。

### 7.2 卡片槽状态

空状态：

- 显示提示文案：把地图上的卡片拖到这里，开始规划常熟路线。

有内容状态：

- 显示路线名称。
- 显示已选地点数量。
- 显示卡片列表。
- 显示路线操作按钮。

P 人生成状态：

- 显示系统生成的路线主题。
- 例如：文化慢游线、美食随走线、非遗体验线。

## 8. J 人 / P 人规划功能

### 8.1 J 人模式

J 人模式强调用户自己规划。

第一版本功能：

- 用户自己点击地图点位。
- 用户自己查看卡片。
- 用户自己拖动卡片进入行程槽。
- 用户自己调整行程顺序。
- 地图根据右侧顺序显示路线连接线。

可选增强：

- 地图进入绘制模式。
- 用户可以在地图上手绘路线或圈出想去区域。
- 第一版本可以先放入口，不一定实现复杂绘制。

### 8.2 P 人模式

P 人模式强调系统帮助规划。

第一版本功能：

- 一键生成推荐路线。
- 一键随机生成路线。
- 生成后仍然使用右侧卡片槽展示。
- 用户可以继续手动调整系统生成结果。

推荐路线类型：

- 景点经典线
- 非遗体验线
- 美食探索线
- 半日轻松线
- 一日综合线

### 8.3 Agent 规划策略

第一版本可以先使用规则模拟 Agent，不一定马上接入真实大模型。

规则规划示例：

- 用户选择「景点经典线」：优先选择 2 到 3 个景点，搭配 1 个美食店铺。
- 用户选择「非遗体验线」：优先选择 1 到 2 个非遗点位，搭配附近景点或美食。
- 用户选择「美食探索线」：优先选择 1 个美食条目，2 到 3 个相关店铺。
- 用户选择「随机路线」：从所有可见点位中随机选取 3 到 5 个。

后续版本可以接入真正 Agent，根据用户时间、预算、偏好、人流量估计、距离等因素生成路线。

## 9. 数据设计

第一版本使用本地静态 JSON 数据。

### 9.1 点位统一数据结构

```ts
type PlaceType = "scenic" | "heritage" | "food" | "restaurant";

type CrowdLevel = "low" | "medium" | "high" | "very-high";

type Place = {
  id: string;
  type: PlaceType;
  name: string;
  subtitle: string;
  summary: string;
  tags: string[];
  position: {
    x: number;
    y: number;
    lng?: number;
    lat?: number;
  };
  openTime?: string;
  price?: string;
  crowdLevel?: CrowdLevel;
  duration?: string;
  history?: string;
  detail?: string;
  suitableFor?: string[];
  notice?: string;
  foodProfile?: FoodProfile;
  restaurantProfile?: RestaurantProfile;
};

type FoodProfile = {
  flavor: string;
  history: string;
  recommendedScene: string;
  relatedRestaurants: string[];
};

type RestaurantProfile = {
  mainFoods: string[];
  averageCost: string;
  popularity: CrowdLevel;
  reviewSummary: string;
  recommendedDishes: string[];
  queueTip: string;
};
```

### 9.2 示例数据说明

V1 至少准备 12 到 16 个点位：

- 景点：4 到 5 个
- 非遗：3 到 4 个
- 具体美食：3 到 4 个
- 美食店铺：4 到 5 个

具体开放时间、门票价位、点评、人流估计先以示例数据填充，并在数据文件中标记：

```ts
dataStatus: "demo" | "verified"
```

其中：

- `demo` 表示演示数据。
- `verified` 表示已经人工核验。

第一版本默认使用 `demo` 数据。

## 10. 推荐技术方案

### 10.1 前端框架

推荐使用：

- Vite
- React
- TypeScript

原因：

- 启动快。
- 组件化适合卡片系统。
- TypeScript 适合维护不同类型卡片数据。
- 便于后续接入地图、拖拽、Agent 接口。

### 10.2 地图与交互

可选库：

- Leaflet：用于真实地图与 Marker。
- 自定义 SVG / div 地图：用于展示型地图。
- dnd-kit：用于卡片拖拽与排序。
- lucide-react：用于按钮和类型图标。

V1 推荐组合：

- React + TypeScript
- 自定义地图容器
- dnd-kit
- lucide-react

如果后续需要真实地图，再将地图层替换为 Leaflet。

### 10.3 样式

推荐：

- CSS Modules 或普通 CSS
- 不优先引入复杂 UI 框架
- 使用清晰的 CSS 变量管理颜色、间距、阴影

页面风格建议：

- 工具型界面，不做宣传页。
- 地图区域宽阔。
- 右侧规划区域清晰。
- 卡片信息密度适中。
- 不使用过多装饰。

## 12. 核心状态设计

前端需要维护以下状态：

```ts
type PlannerMode = "j" | "p";

type AppState = {
  mode: PlannerMode;
  activeFilters: PlaceType[];
  selectedPlaceId: string | null;
  expandedPlaceId: string | null;
  itineraryIds: string[];
  generatedRouteName: string | null;
};
```

状态说明：

- `mode`：当前是 J 人模式还是 P 人模式。
- `activeFilters`：当前显示哪些类型点位。
- `selectedPlaceId`：当前点击的地图点位。
- `expandedPlaceId`：当前正在详细展示的卡片。
- `itineraryIds`：右侧行程槽中的地点顺序。
- `generatedRouteName`：P 人生成路线时的路线名称。

## 13. 关键交互流程

### 13.1 点击点位查看卡片

流程：

1. 用户打开页面。
2. 地图显示所有点位。
3. 用户点击某个点位。
4. 地图附近弹出简化卡片。
5. 用户可以拖动卡片，也可以点击加入按钮。

### 13.2 拖拽规划路线

流程：

1. 用户点击地图点位。
2. 用户把弹出的卡片拖入右侧行程槽。
3. 行程槽出现该卡片。
4. 地图点位高亮。
5. 地图上显示路线序号。
6. 多个地点加入后，地图用线连接。

### 13.3 双击卡片展开

流程：

1. 用户双击卡片。
2. 卡片进入详细模式。
3. 详细卡片宽度和高度增加。
4. 内容区可下拉查看。
5. 用户点击收起后回到简化状态。

### 13.4 P 人生成路线

流程：

1. 用户切换到 P 人模式。
2. 用户点击「生成推荐路线」或「随机路线」。
3. 系统根据规则选择点位。
4. 右侧行程槽自动填入卡片。
5. 地图上同步显示路线点位和连线。
6. 用户可以继续删除、拖动排序、加入新地点。

## 14. 第一版本开发步骤

### 第一步：项目初始化

- 创建 Vite + React + TypeScript 项目。
- 配置基础样式。
- 建立数据、组件、类型目录。

产出：

- 项目可以本地启动。
- 页面有基础布局。

### 第二步：完成地图静态展示

- 实现常熟地图区域。
- 放置示例点位。
- 不同类型点位有不同颜色和图标。
- 支持点位筛选。

产出：

- 页面进入后能看到常熟地图和点位。

## 15. 第一版本验收标准

V1 完成后，需要满足以下条件：

1. 打开页面后，第一眼看到的是常熟市地图。
2. 地图上至少有 12 个点位。
3. 点位至少覆盖景点、非遗、美食、美食店铺四类。
4. 点击任意点位都能展示对应卡片。
5. 景点卡片包含开放时间、门票价位、人流量估计、历史简介。
6. 非遗卡片包含开放时间、体验价位、人流量估计、历史简介。
7. 具体美食卡片包含美食历史简介。
8. 具体美食店铺卡片包含火热情况估计和店铺点评。
9. 卡片可以拖动或加入右侧卡片槽。
10. 右侧卡片槽可以显示规划路线。
11. 右侧卡片槽支持调整顺序。
12. 双击卡片可以进入详细模式。
13. 详细卡片可以滚动查看内容。
14. J 人模式可以手动规划。
15. P 人模式可以生成或随机生成路线。
16. P 人生成路线后，用户仍然可以手动修改。
17. 地图上可以看出已选地点和路线顺序。

## 17. 风险与处理方式

### 17.1 地图准确性风险

风险：

如果使用自定义地图，点位位置不是严格真实坐标。

处理：

V1 将其定位为展示型规划地图，并在数据结构中保留经纬度字段，方便后续替换真实地图。

### 17.2 数据真实性风险

风险：

开放时间、门票、人流、点评可能不准确或过期。

处理：

V1 使用演示数据，并标记 `dataStatus`。后续版本再人工核验或接入真实数据源。

### 17.3 拖拽交互复杂度风险

风险：

拖拽跨区域、排序、移动端兼容会增加开发复杂度。

处理：

V1 同时提供「拖拽」和「加入行程」按钮。即使拖拽在某些设备上体验不佳，用户仍可完成规划。

### 17.4 P 人 Agent 预期风险

风险：

用户可能以为 P 人模式是真正 AI 规划。

处理：

V1 使用规则模拟 Agent，界面文案可以写成「智能推荐」或「路线生成」。后续版本再接入大模型。
