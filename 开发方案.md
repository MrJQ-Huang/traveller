# 常熟地图旅行规划 Web 开发方案

## 0. 文档说明

本文档用于指导「常熟地图旅行规划 Web」从第一版 Demo 继续完善为一个可展示、可迭代、可扩展的产品原型。

当前项目已经完成第一版 Demo，核心体验已经跑通：

- 打开页面直接进入常熟市地图。
- 地图上展示景点、非遗、美食、美食店铺四类点位。
- 点击点位后展示卡片。
- 卡片可以加入或拖入右侧行程槽。
- 右侧行程槽可以形成路线顺序。
- 地图根据行程顺序高亮点位并连线。
- 支持 J 人手动规划。
- 支持 P 人推荐路线和随机路线。
- 支持双击卡片展开详细内容。

本文档接下来要做的事情，是把「已经能跑的 Demo」整理成更完整的产品与技术方案，方便后续继续开发、答辩展示、团队协作和版本迭代。

## 1. 项目定位

### 1.1 一句话定位

一个以常熟市地图为入口的本地文化旅行规划 Web，用户可以在地图上发现景点、非遗、美食和店铺，通过卡片理解内容，并把卡片拖入右侧行程槽来规划自己的常熟路线。

### 1.2 产品关键词

- 常熟地图
- 本地文化
- 景点发现
- 非遗体验
- 美食探索
- 卡片式信息
- 拖拽式行程规划
- J 人手动规划
- P 人自动生成
- 可展示 Demo
- 可扩展数据底座

### 1.3 产品核心闭环

```text
进入地图
  -> 发现点位
  -> 点击点位
  -> 查看卡片
  -> 拖入行程槽
  -> 调整路线
  -> 查看地图连线
  -> 得到自己的常熟路线
```

P 人模式闭环：

```text
进入地图
  -> 切换 P 人模式
  -> 选择路线主题或随机生成
  -> 系统填充行程槽
  -> 用户继续微调
  -> 得到系统辅助生成的常熟路线
```

### 1.4 产品差异点

本项目不只是一个旅游列表页，也不只是一个地图点位页。它的差异点在于：

1. 地图是入口，用户先看到的是常熟空间，而不是信息列表。
2. 卡片是信息容器，景点、非遗、美食、店铺都统一用卡片承载。
3. 右侧卡片槽是规划容器，用户可以像整理便签一样整理路线。
4. J 人和 P 人是规划方式差异，既支持自己安排，也支持系统生成。
5. 常熟本地文化内容是核心主题，不做泛化旅游平台。

## 2. 目标用户与使用场景

### 2.1 目标用户

第一类：来常熟短途旅行的人

- 需求：快速知道常熟有什么值得去。
- 痛点：景点、美食、非遗信息分散，不知道怎么组合。
- 产品价值：用地图和卡片快速形成路线。

第二类：本地或周边城市周末出行用户

- 需求：半日或一日轻松游。
- 痛点：熟悉城市但缺少新的组合玩法。
- 产品价值：P 人路线生成可以提供灵感。

第三类：文化体验型用户

- 需求：想了解常熟的历史、园林、非遗、美食故事。
- 痛点：普通地图只告诉位置，不解释文化意义。
- 产品价值：详细卡片展示历史简介和体验建议。

第四类：黑客松评委和展示观众

- 需求：短时间看懂项目价值和完整度。
- 痛点：很多 Demo 只有概念，没有可操作体验。
- 产品价值：地图、卡片、拖拽、路线生成都能现场演示。

### 2.2 典型使用场景

场景一：J 人手动规划

用户打开页面后，先筛选「景点」和「美食店铺」，点击虞山、方塔园、蕈油面馆等点位，查看卡片后拖入右侧行程槽，再调整顺序，形成自己的半日路线。

场景二：P 人自动生成

用户不知道怎么安排，切换到 P 人模式，选择「文化慢游线」或「美食探索线」，点击生成。系统自动把相关卡片放入右侧行程槽，用户再删掉不感兴趣的地点，加入新的店铺。

场景三：非遗主题体验

用户筛选「非遗」，查看花边体验、糕团制作、评弹茶座等点位。双击卡片看详细介绍，了解体验价格、开放时间和历史简介，然后拖入行程槽形成一条非遗路线。

场景四：答辩展示

展示者打开页面，先展示地图全貌，再点击点位出现卡片，拖入右侧行程槽，切换 P 人模式生成路线，最后双击卡片展示详细内容，说明后续可以接入真实数据和大模型 Agent。

- 不做登录、收藏同步、跨设备保存。
- 不做真实购票、预约、支付。
- 不做复杂后台管理系统。
- 不做真实道路级导航。
- 不做公交、驾车、步行导航算法。
- 不做实时人流数据接入。
- 不做实时点评平台接入。
- 不做完全准确的 POI 商业数据库。
- 不保证 Demo 数据的营业时间、价格、点评完全真实。

## 5. 页面与交互设计

### 5.1 顶部工具栏

顶部工具栏承担全局控制功能。

必须包含：

- 项目名称或品牌标识。
- 数据状态提示，例如「Demo 数据，开放时间与价格待核验」。
- 类型筛选按钮：景点、非遗、美食、店铺。
- J 人 / P 人模式切换。
- 路线主题选择。
- 生成路线按钮。
- 随机路线按钮。
- 当前行程站点数量。
- 预计游玩时长。
- 清空行程按钮。

交互规则：

- 点击类型筛选按钮后，地图只显示对应类型点位。
- 至少保留一种类型被选中，避免地图完全空白。
- 点击 J 人模式后，用户以手动拖拽和手绘为主。
- 点击 P 人模式后，用户以生成路线为主，但仍可手动修改。
- 清空行程只清空右侧行程和地图路线，不删除原始点位。

### 5.2 地图区域

地图区域是产品第一视觉中心。

当前 V1 使用 Leaflet 地图方案：

- 地图中心定位到常熟附近。
- 使用在线地图瓦片。
- 点位使用本地静态经纬度数据。
- 路线连线使用点到点折线，不代表真实道路。

后续可扩展方案：

- 替换为高德地图、百度地图或 Mapbox。
- 使用真实 POI 坐标。
- 接入步行、驾车、骑行路径规划。
- 使用自定义常熟地图底图，强化展示风格。

地图点位状态：

- 默认：显示小坐标图标。
- 悬浮：显示地点名称。
- 点击：弹出地点卡片。
- 已加入行程：点位高亮。
- 已加入行程且有顺序：点位显示序号。
- 被筛选隐藏：不显示 Marker。

地图路线状态：

- 行程中少于 2 个地点：不显示路线折线。
- 行程中 2 个及以上地点：按右侧卡片顺序连线。
- 右侧顺序变化：路线重新绘制。
- 地点被删除：路线同步更新。

### 5.3 卡片弹出层

点击地图点位后，在地图区域中出现卡片弹层。

卡片弹层规则：

- 弹层默认展示简化卡片。
- 弹层内卡片可以拖拽到右侧行程槽。
- 弹层内卡片可以点击「加入」按钮加入行程。
- 双击卡片或点击「详情」进入详细模式。
- 已加入行程的卡片不重复加入。
- 点击关闭按钮关闭弹层。

### 5.4 右侧行程卡片槽

右侧行程槽承担「规划结果」功能。

必须支持：

- 空状态提示。
- 接收地图卡片拖入。
- 展示已选地点。
- 显示地点顺序。
- 拖拽调整顺序。
- 删除单个地点。
- 清空全部地点。
- 显示路线名称。
- 显示路线描述。
- 显示预计游玩时长。

行程槽状态：

空状态：

```text
尚未选择地点
J 人手动排，P 人一键排。
```

手动规划状态：

```text
手动规划路线
已选地点会按顺序显示在这里。
```

P 人生成状态：

```text
文化慢游线
老城园林、名人故居与虞山山水组合。
```

### 5.5 J 人模式

J 人模式强调「自己规划」。

功能要求：

- 用户自己点击地图点位。
- 用户自己查看卡片详情。
- 用户自己把卡片加入行程槽。
- 用户自己调整行程顺序。
- 地图实时显示路线连线。
- 可使用地图手绘层做路线标记。

J 人模式的产品表达：

```text
用户拥有最大控制权，系统只提供地图、信息和工具。
```

### 5.6 P 人模式

P 人模式强调「系统给灵感」。

功能要求：

- 支持预设路线生成。
- 支持随机路线生成。
- 生成路线后自动填充右侧行程槽。
- 生成路线后地图同步高亮和连线。
- 用户仍然可以拖拽调整或删除卡片。
- 用户可以继续手动加入新地点。

V1 预设路线：

- 文化慢游线
- 非遗体验线
- 美食探索线
- 半日轻松线
- 一日综合线

P 人模式的产品表达：

```text
系统先给一个可用方案，用户再按心情修改。
```

### 5.7 双击展开详情

卡片需要支持简化状态和详细状态。

触发方式：

- 双击卡片。
- 点击「详情」按钮。

收起方式：

- 再次双击卡片。
- 点击「收起」按钮。

详细状态要求：

- 卡片宽度或高度明显增加。
- 内容区域可滚动。
- 展示更完整的历史、点评、价格、人流和规划建议。
- 在移动端也应提供按钮入口，不能只依赖双击。

## 7. 数据设计

### 7.1 类型定义

```ts
type PlaceType = "scenic" | "heritage" | "food" | "restaurant";

type CrowdLevel = "low" | "medium" | "high" | "very-high";

type DataStatus = "demo" | "verified";

type PlannerMode = "j" | "p";
```

### 7.2 点位数据结构

```ts
type Place = {
  id: string;
  type: PlaceType;
  name: string;
  subtitle: string;
  summary: string;
  tags: string[];
  position: {
    x: number;
    y: number;
    lng: number;
    lat: number;
  };
  openTime?: string;
  price?: string;
  crowdLevel?: CrowdLevel;
  duration?: string;
  history?: string;
  detail?: string;
  suitableFor?: string[];
  notice?: string;
  foodProfile?: FoodProfile;
  restaurantProfile?: RestaurantProfile;
  dataStatus: DataStatus;
};
```

### 7.3 美食扩展字段

```ts
type FoodProfile = {
  flavor: string;

```

### 7.4 店铺扩展字段

```ts
type RestaurantProfile = {
```

```ts
type RoutePreset = {
```

### 7.6 数据状态规范

所有开放时间、价格、人流量、点评类字段都需要标记数据状态。

```text
demo      演示数据，未核验
verified 经过人工核验的数据
```

页面展示时需要避免把 Demo 数据表达为实时准确数据。

推荐文案：

```text
Demo 数据 · 开放时间与价格待核验
```

### 7.7 数据质量要求

V1 数据最低要求：

- 点位数量不少于 12 个。
- 四类点位都必须出现。
- 景点不少于 4 个。
- 非遗不少于 3 个。
- 具体美食不少于 3 个。
- 美食店铺不少于 4 个。
- 每个点位必须有名称、类型、简介、位置。
- 景点和非遗必须有开放时间、价格、人流估计、历史简介。
- 美食必须有历史简介。
- 店铺必须有火热情况估计和点评。

后续数据增强要求：

- 补真实经纬度。
- 补真实开放时间。

## 8. 路线规划设计

### 8.1 路线生成原则

第一版路线生成不是严格的交通路径算法，而是「点位组合算法」。

重点是：

- 让用户快速得到一组合理地点。
- 让路线看起来有主题。
- 让用户仍然可以修改路线。
- 让答辩时能展示 Agent 规划雏形。

### 8.2 J 人路线规则

J 人模式不自动改动用户路线。

用户操作优先级：

1. 用户加入地点。
2. 用户拖拽排序。
3. 用户删除地点。
4. 用户手绘补充路线想法。

系统只做：

- 保存顺序。
- 显示序号。
- 连线展示。
- 计算预计时长。

### 8.3 P 人路线规则

P 人模式自动生成初始路线。

路线生成方式：

- 预设路线：从配置文件读取固定 placeIds。
- 随机路线：从当前筛选类型中随机抽取若干点位。
- 后续 Agent 路线：根据偏好、时间、预算、人流和距离生成。

V1 规则：

```text
文化慢游线：老城景点 + 名人文化 + 虞山 + 正餐
非遗体验线：非遗体验点 + 曲艺/手作 + 轻食/伴手礼
美食探索线：具体美食 + 关联店铺 + 正餐
半日轻松线：单个大景区 + 附近餐饮 + 轻食
一日综合线：景点 + 非遗 + 美食店铺混合
```

### 8.4 后续 Agent 规划输入

未来接入真实 Agent 时，用户可输入：

- 出发时间
- 可游玩时长
- 预算
- 交通方式
- 偏好类型
- 是否避开人多
- 是否带老人儿童
- 是否想吃正餐
- 是否需要伴手礼

Agent 输出：

- 路线标题
- 路线说明
- 排序后的点位列表
- 每站推荐停留时长
- 每站推荐理由
- 注意事项
- 可替换地点

## 9. 技术方案

### 9.1 当前技术栈

### 9.2 技术选型理由

Vite：

- 启动快。
- 适合黑客松 Demo。
- 配置轻量。

React：

- 组件化清晰。
- 适合卡片、地图、行程槽等状态驱动界面。

TypeScript：

- 点位数据类型复杂。
- 不同卡片字段需要类型约束。
- 后续接接口时更安全。

Leaflet：

- 地图库轻量。
- Marker 和 Polyline 能满足 V1 需求。
- 不强依赖商业地图 Key。

普通 CSS：

- 控制直接。
- 避免第一版引入过重 UI 框架。
- 便于根据展示效果快速调整。

## 10. 视觉与体验规范

### 10.1 页面气质

页面应当是工具型、规划型、文化型。

应该避免：

- 纯宣传首页。
- 大段口号式文案。
- 过度装饰。
- 信息卡片过度营销化。
- 只有颜色没有层次的单调界面。

应该强调：

- 地图第一。
- 卡片清楚。
- 操作直接。
- 路线结果明确。
- 内容有常熟文化感。

### 10.2 类型色彩

建议色彩语义：

```text
景点：绿色，代表山水和户外
非遗：蓝色，代表文化和工艺
美食：橙色，代表味觉和食物
店铺：红色，代表热度和消费场景
```

卡片必须：

- 标题清晰。
- 类型标签明显。

- 详细内容可滚动。

- 不因内容变化导致布局剧烈跳动。
- 在右侧栏中保持可扫读。

- 页面不横向溢出。
- 地图可正常显示。

## 13. 风险与解决方案

### 13.1 数据真实性风险

风险：

开放时间、票价、人流和点评可能不准确。

解决：

- V1 标注 Demo 数据。
- 数据结构加入 `dataStatus`。
- V1.1 增加数据来源和更新时间。
- 答辩时明确当前为原型数据。

### 13.2 地图服务风险

风险：

在线瓦片可能加载慢或受网络影响。

解决：

- V1 保持 Leaflet 轻量方案。
- 准备自定义常熟地图底图作为备用。
- 后续可切换到商业地图服务。

### 13.3 拖拽兼容风险

风险：

桌面端拖拽体验较好，移动端原生拖拽不稳定。

解决：

- 保留「加入」按钮作为替代路径。
- 移动端后续增加底部操作按钮。
- V1.2 可引入专门拖拽库。

### 13.4 P 人预期风险

风险：

用户可能以为 P 人模式已经是真正 AI Agent。

解决：

- V1 表达为「路线生成」或「智能推荐雏形」。
- 文档中说明当前是规则和预设路线。
- V2 再接入真实 Agent。

### 13.5 内容过载风险

风险：

卡片内容过多会影响扫读。

解决：

- 简化卡片只保留关键信息。
- 详细卡片滚动展示。
- 长文案拆成历史、规划建议、注意事项等模块。
